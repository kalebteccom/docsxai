<div class="docsx-section not-content">
  <h2 class="docsx-section-title">Why it exists</h2>
  <div class="docsx-section-sub">
    Screenshot docs rot the moment the UI moves. docsxai treats them as a
    build artifact: a written flow is the source, the rendered doc pack is
    the output, and a re-run is the refresh.
  </div>

  <div class="docsx-grid">
    <div class="docsx-card">
      <span class="docsx-card-mark">// execution</span>
      <h3>Deterministic by contract</h3>
      <div class="docsx-card-text">
        A frozen clock, pinned locale, timezone, and viewport: the same flow
        against the same target state produces byte-identical screenshots,
        keystone-enforced against real Chromium.
      </div>
    </div>
    <div class="docsx-card">
      <span class="docsx-card-mark">// calibration</span>
      <h3>AI-assisted, once</h3>
      <div class="docsx-card-text">
        An agent helps you author the flow-file one time, through the Claude
        Code plugin or any MCP host. After that, every run is agent-free:
        no model in the loop, no token bill per refresh.
      </div>
    </div>
    <div class="docsx-card">
      <span class="docsx-card-mark">// output</span>
      <h3>Annotated screenshots</h3>
      <div class="docsx-card-text">
        Halos, numbered badges, and callouts render from annotations.json
        over clean captures. Redactions mask sensitive UI before any PNG
        hits disk, so the pack is safe to share.
      </div>
    </div>
    <div class="docsx-card">
      <span class="docsx-card-mark">// pipeline</span>
      <h3>CI-friendly publishing</h3>
      <div class="docsx-card-text">
        Re-run on change in your pipeline, then publish the pack as a static
        viewer, an Astro Starlight site, or straight to Confluence through
        the publisher plugin.
      </div>
    </div>
  </div>
</div>
